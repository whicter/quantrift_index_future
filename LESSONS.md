**时间跨度**：2026年5月 ~ 6月
**系统**：NQ + ES 多周期 Confluence 趋势跟踪，IB 实盘

## 一、策略参数优化

### 问题：1H 过度交易
- 优化前：1131 笔交易，最大回撤 -62.9%
- 根因：`use_squeeze_mr=True` 混入了均值回归信号，`ut_key` 太小导致止损过于敏感
- 修复：关掉 `use_squeeze_mr`，`ut_key` 从 0.72 调到 1.0，`min_score` 从 4 升到 5
- 结果：261 笔，回撤降到 -19.6%，胜率 68.6%

### 经验
- 趋势策略和均值回归策略不能混用同一套评分体系——它们对"信号方向"的定义本质不同
- `ut_key` 是最敏感的参数，直接控制止损频率；宁可慢一点也不要太敏感
- 优化目标：先控回撤，再看 Sharpe，不要只看收益率

## 二、Staged TP（分批止盈）的重要性

- 对比测试：staged_tp=True vs False，$100k 起始
- 结果：+$146k vs +$4.5k，相差 32 倍
- 原因：市场经常不能走完整段趋势，先锁定一部分利润大幅提升胜率（45%→56%）
- **结论：分批止盈是这套策略的核心，不可关闭**

## 三、实盘基础设施

### 问题：pm2 开机自启需要 sudo
- `pm2 startup launchd` 生成的命令需要 sudo，在 SSH 非交互式环境里没法用
- 方案：手动写 `~/Library/LaunchAgents/pm2.congrenhan.plist`，`launchctl load` 不需要 sudo
- 经验：macOS 用户级 LaunchAgent 是最干净的开机自启方案，不依赖 sudo 也不依赖 cron

### 问题：launchd stop plist 用 pkill 杀进程
- 原来用 `pkill -f live_engine.py` 强杀，pm2 会立刻把它拉起来，等于没停
- 修复：改成 `pm2 stop ib-bot`，让 pm2 优雅停止并标记为 stopped

### 问题：restart_engine.sh 脚本过期
- 脚本里还是老的 `pkill + nohup venv/python` 命令，跟 pm2 架构完全冲突
- 教训：引入新的进程管理工具后，所有运维脚本必须同步更新

## 四、数据一致性与安全性

### 问题：ContFuture 连续合约 vs 历史 CSV 数据不同
- 回测用 CSV（旧合约价格），实盘用 ContFuture（当前合约），价格有跳空
- 导致：回测末态显示有仓，但当前 Bar 的指标其实不满足入场条件（幻象信号）
- 修复：`do_connect()` 第 4.5 节——空仓时额外校验当前 Bar 指标是否真正满足入场条件

### 问题：重连后状态文件与 IB 实际持仓不一致
- 场景：引擎崩溃期间止损触发，IB 已平仓，但状态文件还记录着持仓
- 重启后：按旧状态计算 delta，可能反向开仓
- 修复：`do_connect()` 重连时对比 IB 净仓和状态文件之和；不一致时自动修正状态文件并发 Telegram 告警
- 修正逻辑：IB=0 全清；IB≠0 让绝对值最大的 TF 吸收差额

### 问题：open orders 没有在重连时检查
- 引擎崩溃时如果有订单在途，重启后不知道订单状态
- 现状：重连时调用 `reqAllOpenOrders()` 打日志，有未平仓单会告警，但不自动处理
- 未来改进：检测到在途订单时应暂停该品种的新信号，等订单结束后再继续

## 五、监控与告警

### 教训：没有告警等于裸奔
- 早期没有任何告警，引擎崩了只能靠手动看日志发现
- 告警触发时机：IB 连接成功/断开、持仓不一致、每笔下单、引擎崩溃重启、周度复盘
- 实现：`tg_alert()` 用线程非阻塞发送，捕获所有异常静默失败，不影响主逻辑

### 教训：state file 是唯一的"内部账本"
- 引擎完全依赖 `live_state.json` 判断当前持仓，IB 只是执行层
- 任何手动操作 TWS 之后，必须同步修改状态文件，否则下次触发会下错单
- 教训：要有一个"状态不一致"的快速检测和告警机制（现在已实现）

## 六、多周期仓位叠加的设计逻辑

### 金字塔建仓：高周期重仓，低周期轻仓
策略没有硬编码手数上限，而是通过 ATR 风险公式自然产生金字塔结构：

```
手数 = (净值 × risk_pct%) / (ATR × sl_mult × 合约乘数)
```

不同 TF 的 `risk_pct_pyramid` 设置不同，但必须确保 risk_pct 的增幅大于 ATR 的增幅，才能产生正确的「高周期手数更多」分层。

**⚠️ 陷阱（2026-06-20 修正）**：原来设置 1H:1.0% / 4H:1.5% / 1D:4.5%，看似高周期风险更高，但高周期 ATR 更大，导致实际手数是 1H 最多（倒置金字塔）。修正后的参数：

| 周期 | risk_pct | ATR(NQ) | 止损金额/手 | 净值 $100k 手数 |
|------|----------|---------|------------|----------------|
| 1H   | **0.72%** | ~80pt  | ~$240      | **3 手**        |
| 4H   | **2.4%**  | ~200pt | ~$600      | **4 手**        |
| 1D   | **6.0%**  | ~400pt | ~$1,200    | **5 手**        |

三 TF 全顺势时最大净仓 = 3+4+5 = **12 手 MNQ**（金字塔顶点）。

**各净值档位对照：**

| NQ/周期 | $32k（统一1%）| $60k（Pyramid）| $100k（Pyramid）| $200k（Pyramid）|
|---------|-------------|--------------|----------------|----------------|
| 1H | 1手 | 2手 | 3手 | 6手 |
| 4H | 1手 | 2手 | 4手 | 8手 |
| 1D | 1手 | 3手 | 5手 | 10手 |
| **最大净仓** | **3手** | **7手** | **12手** | **24手** |

**⚠️ 陷阱：小账户风险失真（1 手最小值 floor）**

公式结果 < 1 手时强制取 1 手，导致小账户高周期的实际风险% 远超意图：

| 周期 | $32k 意图风险(1%) | 实际止损 | 实际风险% |
|------|----------------|---------|---------|
| NQ 1H | $320 | $240 | 0.75% ✅ |
| NQ 4H | $320 | $600 | **1.875%** ⚠️ |
| NQ 1D | $320 | $1,200 | **3.75%** ❌ |
| ES 4H | $320 | $375 | **1.17%** ⚠️ |
| ES 1D | $320 | $750 | **2.34%** ⚠️ |

最坏情况（5 TF 同时触发止损）：-$3,165 ≈ **-9.9%**。不会爆仓，但要做好心理准备，不要因单笔 NQ 1D 亏损 -$1,200 而 panic 干预。此策略设计目标是 **$100k+ 账户**，$32k 下比例失真属过渡期正常现象。

**好处**：
- 高周期趋势更可靠，重仓顺势；低周期噪声多，轻仓试探
- ATR 自适应：波动大时止损宽，自动减仓；波动收缩时自动加仓
- 不需要硬编码上限，市场环境变化时自动调整

### 多周期净仓：内置的动态对冲机制
三个 TF 完全独立持仓，净仓 = 三 TF signed_contracts 之和：

**示例**：4H 多 +4 手，1H 多 +3 手，1D 多 0 手 → 净仓 +7 手

当 1H 出现强烈空头信号（bearScore ≥ reversal_score），1H 执行翻空：
- 关闭 1H 多头 -3 手 + 开 1H 空头 -2 手 → delta = -5
- 净仓从 +7 → +2

**好处**：
- 低周期反转时自动减少风险敞口，不需要人工干预
- 高低周期顺势叠加时自动加重仓位；方向分歧时自动降仓
- 比固定仓位更能适应行情节奏——趋势明确时满仓，转折信号出现时已提前减仓

**实现确认**：代码中每个 TF 独立调用 `process_tf()`，各自计算 delta 并下单；`allow_reversal_flip=True` 时翻转信号直接一笔完成平多+开空。

## 七、架构选型

### 回测重放法 vs 实时信号法
- 选择了回测重放法：每根 Bar 收盘后重新运行完整回测，读取最终持仓
- 优点：实盘逻辑和回测逻辑完全一致，不存在"实盘和回测信号不同"的问题
- 缺点：每次触发要跑完整回测，延迟约 10-30 秒；ContFuture 数据可能和历史 CSV 有价差

### 三周期独立持仓 vs 主周期信号
- 三个 TF 独立计算、独立持仓、仓位叠加
- 优点：高低周期同向时自动加仓，反向时自动对冲降风险
- 风险：三个 TF 同时持仓时总风险敞口放大 3 倍，净值较小时要注意

### pm2 + launchd vs crontab vs Docker
- 当前：pm2（进程守护）+ launchd（开机自启 + 开收市调度）
- Docker 方案已规划但未实施，等系统稳定后再做
- Docker 的主要价值：IBC 自动登录 IB Gateway，彻底消除每周手动操作

## 七、代码质量

### 状态文件格式迁移
- 旧格式：`{"1h": ..., "4h": ..., "1d": ...}`（单品种）
- 新格式：`{"NQ": {"1h": ..., ...}, "ES": {...}}`（多品种）
- 做了向后兼容的自动迁移，但要注意：格式变更时所有使用状态文件的代码都要同步修改

### 全局副作用：`_CaptureStrategy` 类变量
- 策略参数通过类变量传递（`_CaptureStrategy.min_score = ...`），是全局共享状态
- 在三个 TF 串行执行时没问题；如果未来改为并行执行，会有竞态条件
- 现状可接受，但要知道这个设计缺陷存在

## 八、ES 策略设计（2026-06-20）

### ES 1H 趋势策略不可行

- ES 1H Sharpe = -0.445，经 30 轮优化仍为负，永久移除
- 根因：ES 由 500 只股票构成，行业分散，1H 周期震荡特性强，趋势跟踪无效
- ES 4H 和 1D 趋势策略有效（Sharpe 1.025 / 0.657），应保留

### 均值回归（MR）策略关键经验

- **CI > 55 过滤器与信号互斥**：价格到达极端偏离时（RSI<28 + BB下轨），市场往往是短暂趋势下冲，CI 反而降低（变得定向）。CI 过滤几乎屏蔽了所有有效信号（2年只有4个）。移除 CI 过滤。
- **min_score=2 vs 3 天壤之别**：min_score=2（221笔）Sharpe=-1.475；min_score=3（15笔）Sharpe=0.723。宁可每月只做 1 笔，也要确保 3 个信号全中。
- **空头 MR 在牛市无效**：ES 2024-2026 牛市，空头 MR 胜率 38%（多头 55%）。只做多是正确选择，等牛熊切换再评估空头。
- **VWAP 计算需绑定 DatetimeIndex**：backtesting.py 的 `self.data.Close` 是整数索引数组，转成 pandas Series 时要传 `index=DatetimeIndex`，否则 groupby 日期分组失败，VWAP 全为 NaN。
- **R:R = 2:1 是最优组合**：tp=2.0 ATR + sl=1.0 ATR 是测试中最优（Sharpe 0.723），更大的 TP（3.0）反而因持仓时间过长使 Sharpe 下降。
- **同日连续多单是风险点**：回测中 2024-12-18 和 2025-10-10 各触发 2 笔亏损，实盘需加每日最多 1 单限制。

### 多 TF 联合优化的陷阱

ES 4H+1D 联合优化 40 轮，最优分数与初始相同（0.639），说明参数互相干扰。改为单独优化 1D（锁定 4H 参数），ES 1D Sharpe 从 0.253 提升到 0.657。

经验：多 TF 联合优化时，参数空间互相干扰。先优化高 TF 锁定，再单独优化低 TF。
